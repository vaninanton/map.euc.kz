# Скилл: /commit

Выполняй шаги строго по порядку. Не пропускай шаги, не группируй несвязанные проверки.

## Шаг 1. Запустить проверки

Выполни последовательно (каждая следующая только после успеха предыдущей):

```bash
npm run lint
```

```bash
npx tsc --noEmit
```

```bash
npm run test
```

Если любая из команд завершилась с ошибкой:

- Покажи ошибку пользователю
- Исправь её (lint — автоматически через `npx eslint --fix` где возможно; tsc / test — анализируй и правь код)
- Перезапусти упавшую команду
- Только после зелёного статуса всех трёх — переходи к шагу 5

**Не используй `--no-verify` и не пропускай проверки.**

## Шаг 2. Проверить покрытие тестами

Проанализируй изменённые файлы:

- Если добавлены/изменены `.ts` / `.tsx` файлы с бизнес-логикой (компоненты, хуки, утилиты, API) — проверь, есть ли соответствующие `*.test.ts` / `*.test.tsx` файлы среди изменений или в проекте.
- Если тесты отсутствуют или не покрывают новый функционал — **спроси пользователя**:
    > «Для следующих файлов не найдены тесты: `<список>`. Написать unit-тесты? Также нужны e2e-тесты (Playwright)?»
- Если пользователь говорит «да» или «напиши тесты» — напиши тесты, затем вернись к шагу 3.
- Если пользователь говорит «нет» или «пропусти» — продолжай.

> Исключения (тесты не нужны): конфиги (`vite.config.ts`, `eslint.config.ts`, `tailwind.config.*`), миграции БД, статические данные (`*.json`), типы без логики, страницы-роутеры без логики.

## Шаг 3. Сформировать commit message

Изучи:

```bash
git diff --cached
git log --oneline -5
```

Правила commit message для этого репозитория:

- Формат: `type(scope): описание на русском`
- Типы: `feat`, `fix`, `refactor`, `test`, `chore`, `docs`, `perf`
- Scope (необязательно): `admin`, `map`, `api`, `auth`, `e2e`, `deps`
- Описание: короткое, imperative, без точки в конце
- Примеры:
    - `feat(admin): drag & drop и лайтбокс в PhotoManager`
    - `fix(map): поднять контролы над таб-баром`
    - `test(admin): покрытие PhotoManager тестами`

Покажи предложенный commit message пользователю и спроси подтверждение (или предложи отредактировать).

## Шаг 4. Обновить CHANGELOG.md

Если файл `CHANGELOG.md` существует в корне проекта — добавь запись в начало файла:

```markdown
## [Unreleased] — YYYY-MM-DD

### Added / Changed / Fixed

- Краткое описание изменения
```

Если файла нет — создай его с этой записью.

Покажи пользователю добавленную запись перед тем как писать файл.

## Шаг 5. Git commit

Добавь файлы в индекс (только изменённые, не `git add -A`):

```bash
git add <конкретные файлы>
```

Создай коммит:

```bash
git commit -m "$(cat <<'EOF'
<commit message>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

Если pre-commit хук упал — исправь ошибку и создай **новый коммит** (не `--amend`).

После успешного коммита покажи: `git log --oneline -1`

## Шаг 6. Push и Pull Request

Спроси пользователя:

> «Коммит создан. Запушить изменения и открыть Pull Request?»

**Если «да» или «пуш»:**

```bash
git push -u origin <branch>
```

Если ветка — `main`/`master`, предупреди: **«Вы пушите прямо в `main`. Уверены?»** и жди подтверждения.

Затем спроси: **«Создать Pull Request на GitHub?»**

Если да:

```bash
gh pr create \
  --title "<commit message без Co-Authored>" \
  --body "$(cat <<'EOF'
## Summary
<bullet-список изменений из CHANGELOG>

## Test plan
- [ ] `npm run lint` — ✅
- [ ] `npx tsc --noEmit` — ✅
- [ ] `npm run test` — ✅
- [ ] Проверено в браузере

🤖 Generated with [Claude Code](https://claude.ai/claude-code)
EOF
)"
```

Верни пользователю **ссылку на PR**.

**Если «нет»** — завершай, сообщи что коммит создан локально.

## Быстрая справка

| Шаг | Действие                                 |
| --- | ---------------------------------------- |
| 1   | Ветка: если main — предложить новую      |
| 2   | Показать список изменений                |
| 3   | Проверить тесты, предложить дописать     |
| 4   | lint → tsc → test (исправить если упало) |
| 5   | Commit message + подтверждение           |
| 6   | Обновить CHANGELOG.md                    |
| 7   | git add + git commit                     |
| 8   | Спросить про push и PR                   |
